本课新概念（L02 没出现过的）

@tool 装饰器
Python 的语法糖。在普通函数上面加一行 @tool，LangChain 就会自动提取函数名、参数类型、docstring，生成一份 LLM 能读的 JSON 说明书。你不用手写任何 API 文档。跟 L02 的区别：L02 的节点是自己写 prompt 让 LLM 分类，L03 的 @tool 让 LLM 知道"你可以调这个函数"。

bind_tools()
把工具列表塞给 LLM。LLM 拿到说明书之后，每次回复要么直接给答案，要么说"我需要调 add 工具，参数 a=3, b=5"。决定权在 LLM，不在你。L02 的 model 只知道聊天，L03 的 model_with_tools 知道聊天 + 调工具。

ToolNode — 预构建节点
LangGraph 自带的一个节点。你不用像 L01/L02 那样自己写 def my_node(state): return {...}，直接 tool_node = ToolNode(tools) 就完了。它会自动读 LLM 的 tool_calls，挨个执行对应的 Python 函数，把结果打包成 ToolMessage 返回。L04 里你会学怎么手写代替它。

add_messages — reducer
这是 L03 最容易踩的坑。普通 dict 更新是覆盖：返回 {"messages": [新消息]} 会把旧消息删掉。add_messages 是追加：新消息追加到已有消息后面。不用它的后果：ToolNode 返回的 ToolMessage 覆盖了 LLM 之前输出的 AIMessage，LLM 丢失上下文，直接报 INVALID_CHAT_HISTORY。这是社区排第一的报错。

Annotated[list, add_messages]
Python 类型注解的一种写法。翻译成人话："这个字段的类型是 list，更新时不要直接覆盖，用 add_messages 规则追加"。如果你之前写 messages: list，Python 只知道它是列表。加上 Annotated + add_messages，LangGraph 才知道怎么正确更新它。

tool_calls
LLM 返回的 AIMessage 上的一个属性。LLM 不调工具时这个属性不存在。LLM 决定调工具时，它是一个列表，每项包含 {"name": "add", "args": {"a": 3, "b": 5}, "id": "call_xxx"}。这就是 ToolNode 执行工具的指令清单。

hasattr(msg, "tool_calls")
Python 内置函数，检查对象有没有某个属性。在这里就是问"LLM 请求工具了吗？"。有 tool_calls → 走 ToolNode 执行工具。没有 → 说明 LLM 已经给了最终答案，结束。

New Concepts (not covered in L02)

@tool decorator
Python syntax sugar. Add @tool above an ordinary function and LangChain automatically extracts the function name, parameter types, and docstring, generating a JSON manual the LLM can read. No hand-written API docs needed. Difference from L02: L02 nodes wrote prompts asking the LLM to classify text; L03's @tool lets the LLM know "you can call this function."

bind_tools()
Passes the tool list to the LLM. Once the LLM has the manual, each reply is either a direct answer or "I need to call add with a=3, b=5." The LLM decides, not you. L02's model only knew how to chat; L03's model_with_tools knows how to chat and how to call tools.

ToolNode — pre-built node
A node that LangGraph ships with. Instead of writing def my_node(state): return {...} like in L01/L02, you write tool_node = ToolNode(tools) and you're done. It reads the LLM's tool_calls, executes each matching Python function, and packages the results as ToolMessages. In L04 you'll learn how to replace it with hand-written code.

add_messages — reducer
The most common pitfall in L03. Normal dict updates are overwrites: returning {"messages": [new_msg]} deletes old messages. add_messages appends: new messages are added after existing ones. Without it, the ToolNode's ToolMessage overwrites the LLM's previous AIMessage, the LLM loses context, and you get INVALID_CHAT_HISTORY. This is the #1 community error.

Annotated[list, add_messages]
A Python type annotation that says "this field is a list, but don't overwrite it. Use the add_messages rule to append instead." If you wrote messages: list, Python only knows it's a list. Add Annotated + add_messages and LangGraph knows how to update it correctly.

tool_calls
An attribute on the AIMessage the LLM returns. When the LLM decides not to use tools, this attribute doesn't exist. When it decides to, it's a list where each entry contains {"name": "add", "args": {"a": 3, "b": 5}, "id": "call_xxx"}. This is the instruction sheet ToolNode follows.

hasattr(msg, "tool_calls")
Python built-in function that checks whether an object has a given attribute. Here it answers "did the LLM request a tool?" Has tool_calls → go to ToolNode and execute. Doesn't → the LLM already gave the final answer, stop.

🧠 心智模型：Tool = 函数 + 说明书

🧠 Mental Model: Tool = Function + Manual

@tool 装饰器做了两件事：① 保留函数可执行 ② 自动生成一段 JSON Schema（工具名、描述、参数类型）嵌入 LLM 请求中。LLM 看到说明书后决定"要不要调、调哪个、传什么参数"。LLM 不执行工具——它只输出一个 tool_calls 列表，由 ToolNode 执行。

# @tool 等价于给函数贴了一张"说明书"给 LLM 看
@tool
def add(a: int, b: int) -> int:
    """加法。两个整数相加。"""
    return a + b

def llm_node(state: AgentState) -> dict:

def should_call_tool(state: AgentState) -> str:
    last = state["messages"][-1]
    if hasattr(last, "tool_calls") and last.tool_calls:
        return "tools"
    return END

import requests
@tool
def web_search(query: str) -> str: """搜索互联网获取信息"""
    try:
        url = f"https://api.duckduckgo.com/?q={query}&format=json"
        r = requests.get(url, timeout=5)

Q1 (L3-Q2) ToolNode 的工作原理是什么？

Q1 (L3-Q2) How does ToolNode work?

点击查看答案Click to reveal

ToolNode 读取 LLM 返回的 AIMessage 中的 tool_calls 列表，逐个调用对应的 Python 函数，将返回值包装成 ToolMessage，追加到 messages 中。它不修改 LLM 的输出——LLM 只"说"要调工具，ToolNode 才"做"。ToolNode reads the tool_calls list from the LLM's AIMessage, calls each corresponding Python function, wraps the return values as ToolMessages, and appends them to the messages list. It doesn't modify the LLM's output — the LLM "says" which tools to call; ToolNode "does" it.

Q2 (L3-Q3) add_messages reducer 为什么在 Tool Calling 中至关重要？

Q2 (L3-Q3) Why is the add_messages reducer critical in Tool Calling?

点击查看答案Click to reveal

没有 add_messages，每次节点返回的 dict 会覆盖整个 messages 列表。LLM 节点返回 AIMessage 后，ToolNode 返回 ToolMessage——后者直接覆盖前者，LLM 丢失了 tool_call_id 的上下文，OpenAI 返回 400 INVALID_CHAT_HISTORY。add_messages 保证消息追加而非覆盖。Without add_messages, each node's returned dict overwrites the entire messages list. After the LLM node returns an AIMessage, the ToolNode returns a ToolMessage — the latter overwrites the former, the LLM loses the tool_call_id context, and OpenAI returns 400 INVALID_CHAT_HISTORY. add_messages ensures messages are appended rather than overwritten.

Q3 (L3-Q5) LLM 同时调用两个工具时，ToolNode 如何处理？

Q3 (L3-Q5) When the LLM calls two tools simultaneously, how does ToolNode handle it?

点击查看答案Click to reveal

ToolNode 顺序执行所有 tool_calls，每个调用的返回值包装为独立的 ToolMessage。如果两个调用互不依赖（如同时查天气+查汇率），顺序执行结果是正确的。如果两个调用有依赖，应该设计为两次独立的 LLM→Tool 循环。并行执行需要 Send API（L11）。ToolNode executes all tool_calls sequentially, wrapping each return value as an independent ToolMessage. If the two calls are independent (e.g., query weather + query exchange rate), sequential execution produces correct results. If the calls have dependencies, design them as two separate LLM→Tool loops. Parallel execution requires the Send API (L11).
