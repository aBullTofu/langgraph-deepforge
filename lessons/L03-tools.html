概念Concept一句话解释One-Line Explanation
@tool 装饰器decoratorPython 语法糖——在函数定义上面加一行 @tool，自动把它变成 LLM 能"看到"的工具。等价于给函数写了一张带参数说明的 JSON 卡片Python syntax sugar — add @tool above a function definition to automatically turn it into a tool the LLM can "see." Equivalent to writing a JSON card with parameter descriptions for the function
bind_tools()把工具列表传给 LLM——告诉它"你有这些能力可用"。LLM 拿到说明书后，在需要时会输出 tool_callsPass the tool list to the LLM — telling it "these are the capabilities you have." The LLM receives the manual and outputs tool_calls when needed
ToolNode （预构建节点）(pre-built node)LangGraph 自带的"自动执行工具"节点——读取 LLM 的 tool_calls，逐个调用对应的 Python 函数，返回 ToolMessage。不需要自己写 for 循环LangGraph's built-in "auto-execute tools" node — reads the LLM's tool_calls, calls each matching Python function one by one, and returns ToolMessages. No need to write your own for loop
add_messages reducerreducer消息列表的"追加规则"。普通 dict 更新是覆盖——新值覆盖旧值。add_messages 是追加——新消息追加到已有消息后面，不删旧的。不用它的后果：ToolMessage 覆盖 AIMessage → LLM 丢失上下文 → 报错The "append rule" for message lists. Normal dict updates are overwrites — new values replace old ones. add_messages is append — new messages are added after existing ones without deleting them. Consequence of not using it: ToolMessage overwrites AIMessage → LLM loses context → error
Annotated[list, add_messages]Python 类型注解语法——意思是"这个字段是 list 类型，用 add_messages 规则更新"。把类型声明和 reducer 绑在一起Python type annotation syntax — means "this field is a list type, updated using the add_messages rule." Binds the type declaration and reducer together
tool_calls 属性attributeLLM 返回的 AIMessage 上的一个属性——是一个列表，每个元素包含它想调的工具名和参数。如果 LLM 直接回复（不调工具），这个属性不存在An attribute on the AIMessage returned by the LLM — a list where each element contains the tool name and parameters it wants to call. If the LLM replies directly (no tool call), this attribute doesn't exist
hasattr(msg, "tool_calls")Python 内置函数——检查一个对象是否有某个属性。这里用来判断"LLM 请求了工具吗？"有 → 走 ToolNode，没有 → 结束Python built-in function — checks whether an object has a given attribute. Used here to determine "did the LLM request a tool?" Yes → go to ToolNode, No → end

🧠 心智模型：Tool = 函数 + 说明书

🧠 Mental Model: Tool = Function + Manual

@tool 装饰器做了两件事：① 保留函数可执行 ② 自动生成一段 JSON Schema（工具名、描述、参数类型）嵌入 LLM 请求中。LLM 看到说明书后决定"要不要调、调哪个、传什么参数"。LLM 不执行工具——它只输出一个 tool_calls 列表，由 ToolNode 执行。

# @tool 等价于给函数贴了一张"说明书"给 LLM 看
@tool
def add(a: int, b: int) -> int:
    """加法。两个整数相加。"""
    return a + b

🗺 代码地图：Tool Agent 全貌

🗺 Code Map: Tool Agent Overview

👤 用户问 "3+5=?"
  ↓
[llm_node] ← 拿着工具说明书思考 → AIMessage(tool_calls=[{add, a:3, b:5}])
  ↓ (条件边 should_call_tool: 有 tool_calls?)
  ├── YES → [tool_node] → 执行 add(3,5) → ToolMessage(result=8)
  │                              ↓ 回到 llm_node
  │                              LLM 看到结果 "8" → AIMessage("结果是 8")
  │                              ↓ 条件边: tool_calls 为空 → END
  └── NO  → END

核心循环：LLM 说要调工具 → ToolNode 执行 → LLM 看结果 → 决定是否再调工具。add_messages reducer 保证消息追加不覆盖。Core loop: LLM requests a tool → ToolNode executes → LLM sees the result → decides whether to call again. The add_messages reducer ensures messages append rather than overwrite.

from langchain.tools import tool
from langgraph.prebuilt import ToolNode
from langgraph.graph.message import add_messages
from langchain.messages import AnyMessage, SystemMessage
import operator

@tool
def add(a: int, b: int) -> int: """加法运算"""; return a + b
@tool
def multiply(a: int, b: int) -> int: """乘法运算"""; return a * b
@tool
def get_weather(city: str) -> str: """查询城市天气"""
    data = {"北京": "晴 25°C", "上海": "多云 28°C"}
    return data.get(city, f"未找到{city}")

tools = [add, multiply, get_weather]
model = init_chat_model("deepseek-chat")
model_with_tools = model.bind_tools(tools)  # ⭐ 关键：把工具说明书传给 LLM# Key: pass the tool manual to the LLM

Step 2：定义 State（⚠️ 必须用 add_messages reducer！）

Step 2: Define State (⚠️ Must use add_messages reducer!)

class AgentState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]  # ⭐ 追加而非覆盖！
    # 如果用普通的 list，ToolNode 返回的 ToolMessage 会覆盖整个对话历史
    # → 导致 OpenAI 400 错误：tool role 前缺少 tool_calls

Step 3：LLM 节点 + 条件边

Step 3：LLM Node + Conditional Edge

def llm_node(state: AgentState) -> dict:
    msgs = [SystemMessage("你是助手，需要计算时调用工具")] + state["messages"]
    return {"messages": [model_with_tools.invoke(msgs)]}

tool_node = ToolNode(tools)  # 预构建节点，自动解析 tool_calls 并执行# Pre-built node, auto-parses tool_calls and executes

def should_call_tool(state: AgentState) -> str:
    last = state["messages"][-1]
    if hasattr(last, "tool_calls") and last.tool_calls:
        return "tools"
    return END

Step 4：组装（LLM ↔ Tools 循环）

Step 4: Assemble (LLM ↔ Tools Loop)

builder = StateGraph(AgentState)
builder.add_node("llm", llm_node); builder.add_node("tools", tool_node)
builder.add_edge(START, "llm")
builder.add_conditional_edges("llm", should_call_tool, {"tools": "tools", END: END})
builder.add_edge("tools", "llm")  # 工具结果返回 LLM，形成循环
graph = builder.compile()

Step 5：测试

Step 5: Test

    r = graph.invoke({"messages": [HumanMessage(q)]})

Bug 1：messages 应该用 Annotated[list[AnyMessage], add_messages]——这是 #1 报错 INVALID_CHAT_HISTORY 的原因。普通 list 会让 ToolMessage 覆盖 AIMessage，对话历史丢失。

Q1 (L3-Q2) ToolNode 的工作原理是什么？

Q1 (L3-Q2) How does ToolNode work?

点击查看答案Click to reveal

ToolNode 读取 LLM 返回的 AIMessage 中的 tool_calls 列表，逐个调用对应的 Python 函数，将返回值包装成 ToolMessage，追加到 messages 中。它不修改 LLM 的输出——LLM 只"说"要调工具，ToolNode 才"做"。ToolNode reads the tool_calls list from the LLM's AIMessage, calls each corresponding Python function, wraps the return values as ToolMessages, and appends them to the messages list. It doesn't modify the LLM's output — the LLM "says" which tools to call; ToolNode "does" it.

Q2 (L3-Q3) add_messages reducer 为什么在 Tool Calling 中至关重要？

Q2 (L3-Q3) Why is the add_messages reducer critical in Tool Calling?

点击查看答案Click to reveal

没有 add_messages，每次节点返回的 dict 会覆盖整个 messages 列表。LLM 节点返回 AIMessage 后，ToolNode 返回 ToolMessage——后者直接覆盖前者，LLM 丢失了 tool_call_id 的上下文，OpenAI 返回 400 INVALID_CHAT_HISTORY。add_messages 保证消息追加而非覆盖。Without add_messages, each node's returned dict overwrites the entire messages list. After the LLM node returns an AIMessage, the ToolNode returns a ToolMessage — the latter overwrites the former, the LLM loses the tool_call_id context, and OpenAI returns 400 INVALID_CHAT_HISTORY. add_messages ensures messages are appended rather than overwritten.

Q3 (L3-Q5) LLM 同时调用两个工具时，ToolNode 如何处理？

Q3 (L3-Q5) When the LLM calls two tools simultaneously, how does ToolNode handle it?

点击查看答案Click to reveal

ToolNode 顺序执行所有 tool_calls，每个调用的返回值包装为独立的 ToolMessage。如果两个调用互不依赖（如同时查天气+查汇率），顺序执行结果是正确的。如果两个调用有依赖，应该设计为两次独立的 LLM→Tool 循环。并行执行需要 Send API（L11）。ToolNode executes all tool_calls sequentially, wrapping each return value as an independent ToolMessage. If the two calls are independent (e.g., query weather + query exchange rate), sequential execution produces correct results. If the calls have dependencies, design them as two separate LLM→Tool loops. Parallel execution requires the Send API (L11).
