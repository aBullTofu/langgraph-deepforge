📋 Clear Condition

手动实现完整的 ReAct 循环——不依赖 ToolNode，手写 think→act→observe 循环，包含防死循环和完整错误处理。

本课新概念（L03 没出现过的）

手写 act() vs 用 ToolNode
L03 你用的是 ToolNode——一行代码，背后自动执行所有工具，方便但控制不了任何细节。L04 你要自己写 act() 函数替换它。手写之后你可以加权限检查（"这个工具只有管理员能调"）、记录每次调用耗时、截断过长的结果、工具失败后给 LLM 发错误信息让它重试。一句话：简单场景用 ToolNode，需要控制的场景手写 act。

tool_call_id
LLM 发起 tool call 时会生成一个唯一 ID。你创建 ToolMessage 时必须把这个 ID 带上：ToolMessage(result, tool_call_id=tc["id"])。不带的后果：LLM 拿到结果但不知道是哪个调用的回复，直接报错。把它想象成快递单号就对了。

MAX_CALLS — 语义层防循环
在 State 里放一个计数器 tool_calls: int，每次 act() 后 +1。条件边判断超限就强制 END。这是你控制的——"工具最多调 5 次"。跟下面那个物理层的区别：你设 MAX_CALLS=5 是因为你知道 Agent 合理范围内就该完成任务了，不需要更多调用。

recursion_limit — 物理层防循环
LangGraph 编译时的一个参数（默认 25）。图执行超过 25 步直接抛异常，不管你逻辑上是不是还在正常跑。它是最后的保险丝，不是你该依赖的唯一防线。MAX_CALLS 是"你应该在 5 步内搞定"，recursion_limit 是"万一你没搞定，25 步后强制拉闸"。

act() 中的 try/except
工具会失败——网络超时、参数类型不对、第三方 API 挂了。L03 的 ToolNode 失败就失败了，你控制不了。手写 act() 之后，你把工具调用包在 try 里，出错后 catch 住，把错误信息当成正常字符串返回给 LLM："这个工具调用失败了，原因是 xxx，换个方式试试看"。LLM 拿到错误信息后可以调整策略，而不是直接崩溃。

New Concepts (not covered in L03)

Hand-written act() vs ToolNode
L03 used ToolNode: one line of code, auto-executes everything behind the scenes. Convenient but zero control over details. L04 has you write act() yourself. With hand-written code you can add permission checks, log call durations, truncate long results, and return errors to the LLM so it can retry. Short version: ToolNode for simple cases, hand-written act() when you need control.

tool_call_id
The LLM generates a unique ID for each tool call. You must attach it when creating the ToolMessage: ToolMessage(result, tool_call_id=tc["id"]). Without it, the LLM gets the result but doesn't know which call it's responding to, and it errors out. Think of it as a tracking number.

MAX_CALLS — semantic loop guard
Put a counter in State (tool_calls: int), increment after each act(). The conditional edge forces END when the limit is exceeded. This is under your control: "no more than 5 tool calls." It's there because you know an Agent should finish within a reasonable number of calls.

recursion_limit — physical loop guard
A LangGraph compile parameter (default 25). If the graph executes more than 25 steps, it throws GraphRecursionError whether your logic is still running fine or not. It's the last-resort circuit breaker, not your only defense. MAX_CALLS says "finish in 5 steps"; recursion_limit says "if you somehow don't, I'm pulling the plug at 25."

try/except in act()
Tools fail: network timeouts, wrong parameter types, third-party API outages. L03's ToolNode just fails silently, and you have no control. With hand-written act(), you wrap each tool call in try, catch the error, and return the error message as a normal string to the LLM: "This tool call failed because xxx. Try a different approach." The LLM reads the error and adjusts, instead of the whole thing crashing.

🧠 心智模型：三拍子循环

🧠 Mental Model: The Three-Beat Loop

[LLM 思考: 需要查天气] → [执行工具: weather("上海")] → [观察: "28°C"] → [再思考: 可以回复了]

vs 普通代码：顺序执行 A→B→C→结束。ReAct：执行到 B 时可能回到 A，循环到条件满足为止。

def act(state: AgentState) -> dict:
    """手动执行工具——可以加日志、权限检查、结果截断"""
    last_msg = state["messages"][-1]
    tool_msgs = []
    for tc in last_msg.tool_calls:
        print(f"[行动] {tc['name']}({tc['args']})")
        try:
            tool = tools_by_name.get(tc["name"])
            result = tool.invoke(tc["args"]) if tool else f"错误：未找到工具"
        except Exception as e:
            result = f"工具失败：{e}。请调整参数后重试。"  # ⭐ 错误返回给 LLM# Return error to LLM
        print(f"[观察] {result}")
        tool_msgs.append(ToolMessage(str(result), tool_call_id=tc["id"]))
    return {"messages": tool_msgs, "tool_calls": state.get("tool_calls", 0) + len(tool_msgs)}

MAX_CALLS = 5
def should_continue(state) -> str:
    if state.get("tool_calls", 0) >= MAX_CALLS: return END
    last = state["messages"][-1]
    if hasattr(last, "tool_calls") and last.tool_calls: return "act"
    return END

Bug 1：ToolMessage 缺少 tool_call_id——OpenAI/LangChain 需要 tool_call_id 来关联 AIMessage 中的 tool_call 和对应的 ToolMessage。应该用 ToolMessage(str(result), tool_call_id=tc["id"])。

Q1 (L4-Q1) ReAct 和 Plan-Execute 两种 Agent 架构有什么区别？

Q1 (L4-Q1) What's the difference between ReAct and Plan-Execute Agent architectures?

点击查看答案Click to reveal

ReAct 是交替执行：思考→行动→观察→思考，每步都重新评估。Plan-Execute 是先规划后执行：LLM 先列出完整计划，再逐步执行。ReAct 适合不确定性高的场景（需要根据结果调整），Plan-Execute 适合目标明确的任务（写代码、做报告）。LangGraph 中可以用子图实现 Plan-Execute。ReAct is interleaved: think→act→observe→think, re-evaluating at each step. Plan-Execute is plan-first: the LLM first lays out a complete plan, then executes step by step. ReAct suits high-uncertainty scenarios (adapting to results), while Plan-Execute suits tasks with clear goals (writing code, generating reports). In LangGraph, Plan-Execute can be implemented with subgraphs.

Q2 (L4-Q2) LangGraph 的图循环检测机制是什么？

Q2 (L4-Q2) What is LangGraph's graph loop detection mechanism?

点击查看答案Click to reveal

LangGraph 用 recursion_limit（默认 25）限制单次 invoke 的节点执行次数。它是硬限制——不是检测语义上的死循环，而是防止物理上的无限运行。超过限制抛 GraphRecursionError。实际项目中建议双重防护：recursion_limit（硬上限）+ State 中维护调用计数（语义上限，如 MAX_CALLS=5）。LangGraph uses recursion_limit (default 25) to limit the number of node executions per invoke. It's a hard limit — not detecting semantic dead loops, but preventing physical infinite execution. Exceeding the limit raises GraphRecursionError. In practice, use dual protection: recursion_limit (hard ceiling) + a call counter in State (semantic ceiling, e.g., MAX_CALLS=5).

Q3 (L4-Q3) LangGraph 的四种流式模式分别适用什么场景？

Q3 (L4-Q3) Which scenarios suit each of LangGraph's four streaming modes?

点击查看答案Click to reveal

① values：每个节点执行后输出完整 State。适合调试、监控。
② updates：只输出节点返回的增量 dict。适合生产——只关心变化。
③ messages：输出 LLM token 级别流式。适合聊天 UI 的打字效果。
④ debug：输出每个事件的完整信息。适合开发排查。通常用 updates 做生产，messages 做 UI。① values: outputs full State after each node. Best for debugging and monitoring.
② updates: outputs only the incremental dict returned by each node. Best for production — only care about changes.
③ messages: outputs LLM token-level streaming. Best for chat UI typing effects.
④ debug: outputs full event information. Best for development troubleshooting. Typically use updates for production, messages for UI.

Q4 (L4-Q4) interrupt() 和 Command(resume=) 是如何实现 Human-in-the-Loop 的？

Q4 (L4-Q4) How do interrupt() and Command(resume=) implement Human-in-the-Loop?

点击查看答案Click to reveal

interrupt() 在节点中暂停图执行，抛出 GraphInterrupt，把当前状态和提示信息返回给调用方。人在外部审核（修改数据/批准/拒绝），然后调用 graph.invoke(Command(resume=修改后的值), config) 从断点继续。关键：图在 checkpoint 处暂停，恢复时状态不丢失。interrupt() pauses graph execution inside a node, throwing a GraphInterrupt and returning the current state and prompt to the caller. A human reviews externally (modifying data, approving, or rejecting), then calls graph.invoke(Command(resume=modified_value), config) to continue from the breakpoint. Key: the graph pauses at a checkpoint; state is preserved on resume.

Q5 (L4-Q5) LangGraph 的条件边和传统 if/else 路由有什么区别？

Q5 (L4-Q5) What's the difference between LangGraph's conditional edges and traditional if/else routing?

点击查看答案Click to reveal

传统 if/else：编译时固定路径，路由逻辑和业务代码耦合，改规则必须改代码。
LangGraph 条件边：运行时动态路由，路由函数独立可测试，改规则只需改路由函数或 prompt（LLM 路由时），不改业务节点代码。劣势：条件边的映射字典必须包含所有可能返回值，遗漏会导致静默停止。Traditional if/else: compile-time fixed paths, routing logic coupled with business code, changing rules requires changing code.
LangGraph conditional edges: runtime dynamic routing, route functions are independently testable, changing rules only requires changing the route function or prompt (when using LLM routing), without touching business node code. Downside: the conditional edge's mapping dict must include all possible return values — omissions cause silent stops.
