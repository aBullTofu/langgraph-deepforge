A1 · Self-Attention 机制

核心答案

Self-Attention 的核心思想是：让序列中的每个 token 都能直接"看到"序列中的所有其他 token，并根据相关性加权聚合信息。公式为：

Attention(Q, K, V) = softmax(QK^T / √d_k) × V

直觉理解：

  • Q (Query)：当前 token 在"问"："哪些 token 和我相关？"
  • K (Key)：每个 token 在"答"："我是这种类型的 token"
  • V (Value)：每个 token 的"实际内容"，被提取和聚合
  • QK^T：计算注意力分数（相关性矩阵）
  • softmax：归一化为概率分布
  • × V：按注意力权重加权聚合

"Attention Is All You Need" 的含义：Transformer 完全抛弃了 RNN/LSTM 的循环结构，仅用 Attention 机制就实现了序列建模，带来了三个革命性优势：

  1. 并行计算：所有 token 同时处理，无需像 RNN 那样串行
  2. 长距离依赖：O(1) 的路径长度，任意两个 token 之间直接交互
  3. 可解释性：注意力权重可视化了模型"关注"的位置

⭐ 初级能说出 Q/K/V 三个矩阵，给出公式
⭐⭐ 中级能解释除以 √d_k 的原因（防止梯度消失），理解 Multi-Head 的动机（多子空间建模）
⭐⭐⭐ 高级能讨论复杂度 O(n²) 及优化方案（Flash Attention, MQA, GQA），能徒手写出 NumPy 实现

A2 · NumPy 实现 Scaled Dot-Product Attention

标准实现

import numpy as np

def scaled_dot_product_attention(Q, K, V, mask=None):
    """
    Q, K, V: (batch, n_heads, seq_len, d_k)
    mask: (batch, 1, seq_len, seq_len) or None
    """
    d_k = Q.shape[-1]

    # Step 1: 计算注意力分数
    scores = np.matmul(Q, K.transpose(0, 1, 3, 2))  # (batch, heads, seq_q, seq_k)

    # Step 2: 缩放 (关键！防止 softmax 落入梯度饱和区)
    scores = scores / np.sqrt(d_k)

    # Step 3: Masking (如 causal mask 或 padding mask)
    if mask is not None:
        scores = np.where(mask == 0, -1e9, scores)

    # Step 4: Softmax (沿最后一维)
    # 数值稳定版: exp(x - max(x))
    scores_max = np.max(scores, axis=-1, keepdims=True)
    scores_exp = np.exp(scores - scores_max)
    attention_weights = scores_exp / np.sum(scores_exp, axis=-1, keepdims=True)

    # Step 5: 加权聚合
    output = np.matmul(attention_weights, V)

    return output, attention_weights

面试要点

  • 必须做 scores - scores_max 保证数值稳定性，直接用 np.exp(scores) 会在 scores 较大时溢出
  • mask 中 padding 位置填 -1e9 而非 -np.inf，因为 softmax 遇到 NaN 会传播
  • 能讨论 batch matrix multiplication 的维度变换是加分项

⭐⭐ 高级数值稳定版（subtract max）、正确处理 mask、正确的 batch 维度

A3 · 为什么除以 √d_k？

核心答案

原因：防止点积方差随维度增大而增大，导致 softmax 梯度消失。

数学推导：

  • 假设 Q 和 K 的每个元素是独立同分布的，均值为 0，方差为 1
  • QK^T 是 d_k 个独立随机变量的点积和，其方差为 d_k
  • 如果不缩放，当 d_k 很大时（如 64 或 128），QK^T 的值会很大
  • 大的值经过 softmax 后，分布变得极度尖锐（接近 one-hot）
  • softmax 的梯度在尖锐分布下几乎为零 → 梯度消失
  • 除以 √d_k 将方差缩放回 1，使 softmax 输出保持平滑，梯度可以有效传播

如果是 d_k 而不是 √d_k？除以 d_k 会导致方差变为 1/d_k，softmax 过于平滑（接近均匀分布），信息丢失。

面试官评分标准
⭐ 初级"防止内积太大"——知道结果，不知道原因
⭐⭐ 中级能解释方差增大的原因（独立同分布点积）和梯度消失的机制
🏆 高级能推导数学过程，并讨论为什么是 √d_k 而不是 d_k 或其他

A4 · 位置编码（Sinusoidal vs RoPE）

核心答案

Sinusoidal Positional Encoding（Transformer 原始论文）：

PE(pos, 2i)   = sin(pos / 10000^(2i/d_model))
PE(pos, 2i+1) = cos(pos / 10000^(2i/d_model))

优点：可以外推到训练时未见过的序列长度。缺点：绝对位置编码，不直接建模 token 间的相对位置关系。

RoPE (Rotary Position Embedding)（LLaMA, Qwen, DeepSeek 等主流模型使用）：

  • 核心思想：通过旋转矩阵对 Q 和 K 进行编码，使得 QK^T 的结果自然包含相对位置信息
  • 数学本质：对 Q 和 K 的每对维度施加一个旋转，旋转角度与位置 m 成正比
  • 关键性质：RoPE(Q_m, m) · RoPE(K_n, n) = f(Q_m, K_n, m-n)，注意力分数只依赖于相对位置 m-n

RoPE 的优势：

  1. 相对位置建模：天然编码相对位置，符合语言的平移不变性
  2. 外推性好：配合 NTK-aware scaling 或 YaRN 可以很好地外推到更长序列
  3. 与 Attention 优雅结合：旋转变换可以直接融入 QK 计算，不增加额外参数
  4. 递减的远程衰减：相对距离越大，注意力分数自然衰减
面试官评分标准
⭐ 初级知道 Sinusoidal PE 公式，知道 RoPE 是"旋转位置编码"
⭐⭐ 中级能解释 RoPE 的旋转矩阵形式，理解"相对位置"的含义
🏆 高级讨论 NTK-aware / YaRN 外推，知道 MLA (DeepSeek-V2) 如何处理位置编码

A5 · LLM 幻觉的分析与治理

根本原因

  1. 训练数据截止：模型知识有截止日期，无法回答训练后的事件
  2. 概率采样本质：LLM 是 next-token predictor，不是事实数据库。在不确定时仍会"编造"最可能的 token
  3. 缺乏 Grounding：模型没有真实世界的验证机制，无法区分"记忆中的事实"和"生成的错觉"
  4. 训练数据噪声：训练语料本身包含错误信息，模型学习了这些错误
  5. 过度泛化：模型将学到的模式过度应用到不匹配的场景

工程化解决方案

  1. RAG (Retrieval-Augmented Generation)：外挂知识库，检索真实文档作为生成依据。核心：检索到的文档作为 Ground Truth 约束生成
  2. Function Calling + 实时数据：通过 API 获取最新数据（如 Google Search API），而非依赖模型内部知识
  3. Constrained Decoding：使用 grammar/guidance 约束输出格式（如必须从给定选项中选择），从根本上杜绝编造
  4. LLM-as-Judge + Self-Reflection：让 Agent 自我检查输出的 factual consistency，标记可疑内容
  5. Citation/Attribution：要求模型对每个事实声明提供来源引用，无来源的声明标记为"推测"
面试官评分标准
⭐ 初级只说"用 RAG"
⭐⭐ 中级区分原因类型（截止/采样/噪声），给出 2-3 种方案
🏆 高级系统工程视角：检测→溯源→修复→预防的闭环，讨论幻觉率量化

A7 · GPT vs BERT vs T5 架构差异

核心对比

特性GPT (Decoder-only)BERT (Encoder-only)T5 (Encoder-Decoder)
Attention Causal (masked) self-attention Bidirectional self-attention Bidirectional encoder + Causal decoder + Cross-attention
训练目标 Next token prediction (自回归) MLM (Masked Language Modeling) + NSP Span corruption (denoising)
擅长任务 文本生成、对话、代码生成 文本理解、分类、NER、QA 翻译、摘要、改写（seq2seq）

为什么当前 LLM 都是 Decoder-only？

  1. 统一序列建模：所有任务（分类、生成、对话）都可以格式化为"给定前缀预测下一个 token"，一个架构解决所有问题
  2. Scaling Law 友好：自回归目标的数据利用率最高（每个 token 都是训练信号），scaling 行为更好
  3. 上下文学习 (ICL)：Decoder-only 模型的 in-context learning 能力优于 Encoder-Decoder 模型
  4. 工程简单：不需要 Encoder-Decoder 之间的 cross-attention 对齐，实现和优化更简单

A9 · Tokenization 和 BPE

BPE (Byte-Pair Encoding) 原理

  1. 从字符级别开始，每个字符是一个 token
  2. 统计所有相邻 token pair 的出现频率
  3. 将最高频的 pair 合并为一个新 token
  4. 重复 2-3 直到达到目标词汇量（如 50,000）

例如："low" "lower" "newest" "widest" → 统计发现 ("e","s") 高频 → 合并成 "es" → 再发现 ("es","t") 高频 → "est"...

中文 Tokenization 为什么更难？

  1. 无天然分隔符：英文有空格分词，中文是连续字符串，需要先分词
  2. 字符集巨大：常用汉字 6000+，总数数万，远大于英文字母的 26 个
  3. 粒度问题：一个中文词可以由 1-N 个字组成，subword 切分边界不直观
  4. Token 效率低：同样的语义内容，中文所需 token 数通常少于英文（因为一个字密度更高），但更容易出现 OOV

A10 · LLM 输出截断问题排查

诊断框架（从最常见到最罕见）

  1. max_tokens 设置太小：最直接的原因。检查 API 调用的 max_tokens/max_output_tokens 参数
  2. Context Window 超限：输入+输出超出模型的 context window，输出空间被压缩。统计输入 token 数
  3. finish_reason = "length"：检查 API 返回的 finish_reason，确认是因为 token 限制而非自然结束
  4. Output Parser 截断：如果用了自定义 Parser，检查是否在输出未完成时就终止了
  5. 前端截断：模型实际输出了完整内容，但前端/UI 做了截断显示
  6. Streaming 中断：流式传输中连接断开或超时，只展示了部分内容

解决方案

  • 增大 max_tokens（注意成本）
  • 使用 streaming 模式，token 级别输出
  • 实现续写机制：检测到 length 截断后，发送 continuation 请求
  • 压缩输入：摘要历史对话、裁剪无关上下文

A13 · Embedding 基础

核心答案

Embedding 是将文本映射到高维向量空间（通常 768-4096 维），使得语义相似的文本在向量空间中距离接近。

语义空间的特性：

  • 语义聚集：相似含义的文本向量彼此靠近
  • 方向编码：向量方向编码语义特征（如 "king - man + woman ≈ queen"）
  • 线性结构：向量运算在一定程度上对应语义运算

为什么 Cosine Similarity 更常用？

  • 方向敏感，长度不敏感：两个文本的语义相似性更多取决于方向而非向量长度。长文档和短文档如果语义相同，cosine 相似度很高
  • 归一化友好：cosine 相似度范围在 [-1, 1]，容易设定阈值
  • 训练目标对齐：许多 embedding 模型在训练时使用了 cosine similarity 作为损失函数

A14 · 128K Context Window vs RAG

核心答案：长上下文不能替代 RAG

原因：

  1. 成本：128K token 的推理成本是 4K 的 32 倍，且输入 token 通常比输出 token 更贵
  2. 延迟：处理 128K token 的 Attention 计算量是 O(n²)，延迟显著增加
  3. Lost in the Middle：研究表明 LLM 对输入中间部分的信息利用效率远低于开头和结尾。长上下文中，中间的内容几乎被"忽略"
  4. 注意力稀释：无关内容越多，模型越难聚焦到真正相关的信息
  5. 幻觉风险：大量上下文增加了模型混淆不同来源信息的风险

正确的策略：RAG + 适度上下文长度。用 RAG 精确检索相关内容（几百到几千 token），用 LLM 的上下文窗口（8K-32K 足够）处理检索到的内容。

A15 · Prompt Caching

核心答案

Prompt Caching 是 LLM API 提供的一种优化机制：当多个请求共享相同的前缀（如 system prompt、few-shot examples、工具定义）时，API 提供方会缓存这些前缀的 KV cache，后续请求无需重新计算。

工作原理

  1. 第一个请求处理完整 prompt，API 缓存前缀部分的 KV cache
  2. 后续相同前缀的请求直接从缓存读取 KV cache，只计算差异部分
  3. 缓存命中时，这部分 token 的计费大幅降低（如 Anthropic 打 90% 折扣）
  4. 缓存通常在几分钟到几小时内有效

最有效的场景

  • 长的 System Prompt + 短的用户问题（客服、教育、代码助手）
  • 大量 Few-shot examples 放在 prompt 前缀
  • 工具定义（function schemas）放在 system prompt 中
  • 静态的 RAG 检索上下文（知识库很少变化）

设计建议

  • 把静态内容放在 prompt 前面（system prompt + tools）
  • 把动态内容放在 prompt 后面（user message, search results）
  • 使用固定的 breakpoint 标记缓存边界

LangChain 是一个用于构建 LLM 应用的开源框架，提供了标准化的组件和抽象来简化 LLM 应用的开发。

A17 · RAG Pipeline 全过程

完整链路

Ingestion (离线):
  Documents → [Document Loader] → Text
  Text → [Text Splitter / Chunking] → Chunks
  Chunks → [Embedding Model] → Vectors
  Vectors → [Vector Store] → Index

Retrieval + Generation (在线):
  User Query
    → [Query Rewriting / Expansion] → Optimized Query
    → [Embedding Model] → Query Vector
    → [Vector Store Similarity Search] → Top-K Candidate Chunks
    → [Re-Ranker / Filter] → Ranked Relevant Chunks
    → [Prompt Template: System + Context + Query] → Full Prompt
    → [LLM] → Answer

各阶段关键设计决策

  • Document Loader：选型取决于文档格式（PDF, HTML, Markdown, DB...）
  • Chunking：决定检索粒度的最关键环节。考虑 chunk_size 和 overlap
  • Embedding Model：通用场景用 text-embedding-3-large，中文优先考虑 BGE/M3E
  • Vector Store：开发用 Chroma/FAISS，生产考虑 Pinecone/Weaviate/Milvus
  • Re-Ranker：Cross-Encoder 模型（如 Cohere Rerank, BGE-Reranker）提升检索精度

A18 · Naive vs Advanced vs Agentic RAG

类型特点适用场景
Naive RAG 检索→拼接→生成。没有质量控制 简单 QA，对质量要求不高的场景
Advanced RAG + Query Rewriting + Hybrid Search + Re-Ranking + 来源引用 企业知识库、客服系统
Agentic RAG Agent 自主决定何时检索、检索什么、怎么用检索结果。可多轮检索、自我修正 复杂研究任务、需要多源综合的场景

关键升级点：Agentic RAG 不只是"检索+生成"，而是把 RAG 作为一个工具交给 Agent 使用。Agent 可以：判断是否需要检索、选择检索策略、评估检索质量、决定是否重新检索、综合多次检索结果。

A19 · RAG 检索不相关的排查

排查清单（按概率排序）

  1. Chunk Size 不当：太小（失去上下文）或太大（噪音多）。调整 chunk_size 和 overlap
  2. Embedding Model 不匹配：通用 embedding 在专业领域表现差。换领域 fine-tune 的 embedding
  3. Query-Document Gap：用户的查询用词和文档中的用词不同。添加 Query Rewriting（用 LLM 改写/扩展查询）
  4. 检索方式单一：纯向量检索漏掉精确匹配。引入 Hybrid Search（向量+BM25 关键词）
  5. 缺少 Re-Ranker：Top-K 检索召回了一些不太相关的文档。加 Re-Ranker 精细排序
  6. 文档预处理差：PDF 解析乱码、表格被破坏。改进文档解析管道
  7. Metadata 过滤缺失：没有利用文档元数据（时间、来源、类别）做预过滤

A20 · Chunking 策略

Chunk Size 的影响

  • 太小（如 100 字符）：语义不完整，"主轴故障"和"更换方法"被分到不同 chunk，检索时只拿到一半信息
  • 太大（如 2000 字符）：包含过多噪音，干扰 LLM 注意力（Lost in the Middle），且容易超出 context window
  • 最佳实践：通常 500-1000 字符，overlap 100-200 字符。具体取决于文档类型

Semantic Chunking

不按固定长度切分，而是按语义边界切分。方法：

  1. 计算相邻句子的 embedding 相似度
  2. 当相似度骤降（低于阈值）时，标记为语义边界
  3. 在边界处切分

优势：每个 chunk 内部语义连贯，避免在段落中间截断。

其他 Chunking 策略

  • Recursive Character Splitting：按段落→句子→词的优先级递归切分（LangChain 默认）
  • Document-based Splitting：按文档的自然结构（Markdown 标题、HTML 标签）切分
  • Agentic Chunking：用 LLM 决定最佳切分点

A25 · LangChain Memory 类型

主要 Memory 类型：

  • ConversationBufferMemory：存储完整对话历史。简单但 token 消耗随对话增长
  • ConversationBufferWindowMemory：只保留最近 K 轮对话。滑动窗口策略
  • ConversationSummaryMemory：用 LLM 对历史对话做摘要。省 token 但可能丢失细节
  • ConversationSummaryBufferMemory：混合策略——保留最近 K 轮完整对话，更早的做摘要
  • ConversationTokenBufferMemory：按 token 数量限制，而非轮数

LangGraph 的替代方案：在生产中，LangChain 的 Memory 组件逐渐被 LangGraph 的 Checkpointing 替代。Checkpoint 在每个 super-step 自动保存 State，支持更精细的状态管理、时间旅行、持久化。LangGraph 还提供 Store API 处理长期跨会话记忆。

A27 · Re-Ranking

核心答案

Re-Ranking：在向量检索返回 Top-K 候选文档后，使用更精确（但更慢）的模型对候选文档重新排序。

为什么需要？向量检索用的 Bi-Encoder（query 和 document 分别编码）速度快但精度有限。Re-Ranker 用 Cross-Encoder（query 和 document 一起输入）计算相关性分数，精度更高但速度慢。两者组合：Bi-Encoder 粗筛（从百万中召回 Top-50），Cross-Encoder 精排（从 50 中选出 Top-5）。

常用 Re-Ranker：

  • Cohere Rerank：商业 API，效果好，支持多语言
  • BGE-Reranker (BAAI)：开源，中文友好
  • Cross-Encoder (SBERT)：通用方案，可自定义训练
  • ColBERT：Late-interaction，在精度和速度间取平衡

A32 · MCP (Model Context Protocol)

核心答案

MCP 是由 Anthropic 提出的开放协议，标准化了 LLM 应用与外部工具/数据源之间的通信接口。它定义了三个原语：

  • Tools：LLM 可调用的函数（类似 Function Calling，但接口标准化）
  • Resources：LLM 可读取的数据（文件、数据库记录、API 响应）
  • Prompts：预定义的 prompt 模板

与 Function Calling 的关键区别：

  • Function Calling：每个 LLM 提供商有自己的 tool schema 格式（OpenAI vs Anthropic 不兼容）
  • MCP：统一的协议标准，一次定义工具，所有支持 MCP 的 LLM 都可使用
  • MCP 是 Client-Server 架构：MCP Server 暴露工具/资源，MCP Client（LangChain/LangGraph Agent）调用它们
  • 解耦：工具开发者不需要知道最终使用哪个 LLM，LLM 开发者不需要适配每个工具的接口

A36 · LangGraph 核心概念

LangGraph 是 LangChain 生态中的图结构 Agent 编排框架。它把 LLM 工作流建模为有向图（Directed Graph），其中：

  • State：图执行过程中共享的数据对象（TypedDict），所有节点可读写
  • Node：计算单元（Python 函数），接收 State，返回 State 的部分更新
  • Edge：控制流——固定边（确定性）、条件边（动态路由，由函数决定下一个节点）

LangGraph vs LangChain 的核心差异：LangChain 的 Chain 是线性的 A→B→C，LangGraph 支持循环、条件分支、并行执行、状态持久化、人类审批中断——这些都是构建 Agent 的必需品。

A37 · StateGraph 最小 Agent 实现

from typing import Annotated, TypedDict
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages
from langgraph.prebuilt import ToolNode, tools_condition
from langchain_openai import ChatOpenAI
from langchain_core.tools import tool
from langchain_core.messages import BaseMessage

class State(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]  # KEY: add_messages!

@tool
def get_weather(city: str) -> str:
    """Get current weather for a city."""
    return f"Weather in {city}: sunny, 22°C"

tools = [get_weather]
llm = ChatOpenAI(model="gpt-4o-mini").bind_tools(tools)

def chatbot(state: State) -> dict:
    return {"messages": [llm.invoke(state["messages"])]}

graph = StateGraph(State)
graph.add_node("chatbot", chatbot)
graph.add_node("tools", ToolNode(tools))
graph.add_edge(START, "chatbot")
graph.add_conditional_edges("chatbot", tools_condition)  # → "tools" or END
graph.add_edge("tools", "chatbot")

app = graph.compile()
result = app.invoke({"messages": [{"role": "user", "content": "What's the weather in Tokyo?"}]})
print(result["messages"][-1].content)

关键点：add_messages reducer 确保新消息追加到 messages 列表而非覆盖；tools_condition 检查 LLM 响应是否包含 tool_call；ToolNode 自动执行工具并返回 ToolMessage。

A43 · LangGraph Agent Loop (ReAct)

ReAct 循环 = Reasoning + Acting 的交替执行。

在 LangGraph 中体现为：

START → chatbot (Reason: LLM 决定是否使用工具)
          ↓ tools_condition
       ┌─ tools (Act: 执行工具) → 返回 chatbot (Observe: LLM 看到工具结果)
       │
       └─ END (没有工具调用，最终回复)

这是通过 tools_condition 的闭环实现的——只要 LLM 持续返回 tool_call，就会在 chatbot 和 tools 之间循环，直到 LLM 输出纯文本回复。

A44 · Graph 静默退出 5 种原因

  1. 条件边返回 END：tools_condition 或自定义路由函数返回了 END，没有后续节点
  2. 条件边返回值无匹配：路由函数返回了 path_map 中不存在的值，LangGraph 默认行为是停止
  3. 节点返回了不存在的 State key：节点写入了一个未在 State schema 中定义的 key，更新被静默丢弃
  4. Recursion Limit 触发：图执行步数超过 recursion_limit（默认 25），抛出 GRAPH_RECURSION_LIMIT 错误（但可能被静默捕获）
  5. 未配置 checkpointer 时使用 interrupt：interrupt() 需要 checkpointer，否则行为异常

调试方法：开启 LangSmith 追踪，检查每个节点的输入/输出 State。

A52 · 并行 Reducer 重复问题

多个并行节点同时读取 State，各自基于相同的旧 State 做了 append，然后各自写回——Reducer 看到两"批次"更新，每批都包含了"旧数据+新数据"，导致旧数据被重复追加。

A56 · State 设计最佳实践

Small State Principle：State 只存图执行必要的共享数据。原因：

  1. 每次 checkpoint 都会序列化完整 State → State 越大 checkpoint 越慢
  2. 并行节点共享 State → State 越大竞争越激烈
  3. State 通过 checkpoint 持久化 → 大 State 占用更多存储

大对象（如整个知识库）应放在外部存储，State 中只存引用（如 doc_id）。

A61 · Checkpointing 机制

Checkpointing 在每个 super-step 完成后自动保存 State 的快照。它解决了：

  1. 暂停/恢复：Human-in-the-Loop 场景，图可以无限期暂停等待人类
  2. 时间旅行：回退到任意历史 checkpoint 重新执行
  3. 错误恢复：图执行中断后从最近的 checkpoint 继续
  4. 多会话隔离：通过 thread_id 隔离不同用户的状态
  5. 可观测性：完整的状态变更历史用于调试和审计

thread_id 是 checkpoint 的命名空间标识符。每个 thread_id 拥有独立的状态历史和 checkpoint 链。

A17 · DeepSeek MLA 注意力机制

MLA (Multi-head Latent Attention) 是 DeepSeek-V2/V3 提出的注意力优化方案，核心思想：将 Keys 和 Values 投影到低维潜在空间，在潜在空间中进行注意力计算，仅在最后一步展开回原始维度。

工作原理：

  1. 传统 MHA：为每个 head 存储完整的 K 和 V 矩阵 → KV Cache 大小 = 2 × n_layers × d_model × seq_len
  2. MLA：将 K 和 V 压缩到低维潜在向量 c_KV（维度远小于 d_model），缓存只存这个压缩向量
  3. 推理时：从 c_KV 通过上投影矩阵恢复到完整维度，再做 Attention

核心优势：KV Cache 减少 90%+，推理成本大幅降低。这也是 DeepSeek 能做到 GPT-4o 1/10 价格的底层技术原因之一。

对比：MHA (标准) → MQA (多 Query 共享 KV) → GQA (分组共享) → MLA (潜在压缩)，每次迭代都在减少 KV Cache。

A46 · MCP 协议架构原理

MCP (Model Context Protocol) 由 Anthropic 于 2024 年 11 月提出，2025 年 12 月捐给 Linux Foundation Agentic AI Foundation。截至 2026 年，97M+ 下载，被 OpenAI、Google、Microsoft 等主流厂商采用。

架构：

User (Host App)
  └─ MCP Client (内嵌在 Agent 中)
       ├── MCP Server A (文件系统)  → read/write files
       ├── MCP Server B (数据库)    → query DB
       └── MCP Server C (API)       → call external APIs

三个原语：

  • Tools：Agent 可调用的函数（类似 Function Calling）
  • Resources：Agent 可读取的数据源（文件、数据库记录）
  • Prompts：预定义的 prompt 模板

与 Function Calling 的本质区别：FC 每个 LLM 提供商有自己的格式（不兼容），MCP 统一标准 — 一次定义工具，所有支持 MCP 的 LLM 都可用。这解决了 "M×N 集成问题"。

from langchain.agents.middleware import AgentMiddleware

class LoggingMiddleware(AgentMiddleware):
    def before_model(self, state, runtime):
        print(f"[LOG] Messages count: {len(state['messages'])}")
        return None  # None = continue normally

    def on_tool_start(self, state, runtime, tool_call):
        print(f"[TOOL] Calling {tool_call['name']} with {tool_call['args']}")
        return None

2026 年 3 月披露的关键 CVE：CVE-2025-68664（凭据泄露）、CVE-2025-67644（提示注入）。

分层防御体系：

  1. 输入净化层：检测并过滤 prompt injection 模式（分隔符欺骗、角色切换、ignore 指令）
  2. 工具权限分级：只读（search, get）vs 读写（update, create）vs 管理员（delete, config）— 按需授权
  3. MCP Server 验证：安装前审查 MCP Server 源码（类似 npm audit），限制 Server 的网络访问范围
  4. HITL 审批：高风险操作（删除、发送邮件、修改配置）必须人工确认
  5. 审计日志：所有工具调用和状态变更全量记录，支持事后追溯
  6. 凭据管理：API Key 通过 Secrets Manager 注入（非环境变量明文），定期轮换

# langgraph.json (2026 格式)
{
  "graphs": {
    "agent": "./agent.py:graph"
  },
  "env": ".env",
  "python_version": "3.12",
  "checkpointer": {
    "type": "postgres",
    "connection_string": "postgresql://user:pass@host:5432/langgraph"
  },
  "store": {
    "type": "postgres",
    "ttl": {"default_ttl": 1440, "refresh_on_read": true}
  },
  "redis": {
    "url": "redis://localhost:6379"
  }
}

# 部署命令
langgraph deploy --name my-agent --region us-east1

# 开发 vs 生产
langgraph dev          # 本地开发，热重载
langgraph deploy       # 生产部署 (2026.03 新增，替代 langgraph up)
langgraph up           # 旧版 Docker Compose 部署 (仍可用但不推荐)
