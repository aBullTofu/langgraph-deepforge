🏭 水渠与磨坊

L01 · State / Node / Edge / Conditional Edge

古时有一个王国，国王下令沿河修建一条水渠，让河水依次流过五座磨坊。每座磨坊从水渠中取水，完成自己的活计——第一座磨面，第二座织布，第三座榨油，第四座打铁，第五座造纸——然后将水排回水渠，流向下游。

起初一切顺利。直到某天，第四座磨坊的坊主发现上游来的水里混着油渣，他的铁器淬火时出现了裂纹。他派人去查，才发现是第三座榨油坊换了新工艺，排水变脏了。

国王召开磨坊议会。众人争执不下，没人愿意承认是自己的问题。最后国王颁布了一条法令：每座磨坊必须在排水口挂一块木牌，写明自己往水里加了什么、去掉了什么。从那以后，下游的磨坊只要看一眼上游的木牌，就知道水流里有了什么变化。

多年后，国王的孙子继位。他发现水渠有时需要分流——比如大旱之年，灌溉田地的磨坊需要更多水，造纸的磨坊可以暂时停转。他命人在水渠上修了一座分水闸，闸口守卫根据木牌上的信息，决定水流去向："如果水里铁屑过多，说明打铁坊正在全力运转，把水全给铁坊；如果水面漂着纸浆，说明造纸坊需要水，开左闸。"

这个王国繁荣了一千年。后世的历史学家发现，他们发明的木牌流水制和分水闸，正是现代图编程中 State 和 Conditional Edge 的雏形。

🔮 揭示：这是 LangGraph 的 StateGraph 编程范式
水渠→ State（共享数据流）
磨坊→ Node（独立处理单元）
木牌→ State 的字段声明（TypedDict）
水流（木牌+水）→ 节点返回的 dict（部分状态更新）
从上游到下游→ 固定边（add_edge）
分水闸 + 守卫判断→ 条件边（add_conditional_edges + 路由函数）
国王的法令→ 框架约束：节点只能通过 State 通信

🏚 十字路口的客栈

L02 · LLM 节点 + 条件边

丝绸之路上的商队来到一个三岔路口。路口立着一间客栈，客栈里住着一位老旅人。这位老旅人有个神奇的本事：他只要看一眼你的货物和脸色，就能准确告诉你该走哪条路。

"去长安卖丝绸？走左边，官道平坦，税关友好。"

"去波斯卖茶叶？走中间，虽然山路崎岖但路程最短，茶叶不易受潮。"

"去西域卖瓷器？走右边，经敦煌出关，有骆驼商队可以结伴。"

商人问老旅人："你怎么知道该走哪条路？"老旅人笑而不语。直到他临终前才告诉徒弟："我在这里住了四十年，看过十万支商队。他们去了哪条路、后来怎么样，我都记在心里。所以看一眼就明白。"

徒弟继承了客栈。但徒弟没有四十年的经验。于是他想了个办法：给全世界的商人写信——"如果你在这个路口，去长安应该走哪条？去波斯呢？去罗马呢？"——收集每个人的答案，综合起来。虽然没有师父那么准，但答对的概率已经相当高了。

🔮 揭示：LLM 在条件边中的作用 — 用模型的判断能力路由 Agent 的去向
老旅人/徒弟→ LLM 模型
商队的货物和脸色→ 输入（State 中的文本）
三条路→ 条件边的三个可能目标节点
老旅人的经验判断→ LLM 的推理输出
路标指示→ 条件边路由函数返回的字符串

🧰 学徒的工具箱

L03 · Tools / 工具调用

老锁匠收了个徒弟。第一天，他给了徒弟一个上锁的铁盒："打开它。"

徒弟用手掰、用牙咬、用石头砸，折腾了一整天，铁盒纹丝不动。

第二天，老锁匠拿出一串钥匙，一把一把教徒弟："这把开挂锁，这把开暗锁，这把是万能钥匙但有些锁打不开会卡住。"然后他把钥匙挂成一串，递给徒弟："再试。"

徒弟接过钥匙串，对着铁盒试了第二把，"咔嗒"，开了。从此，遇到任何锁，徒弟不再用蛮力——翻看钥匙串，选一把最像的，插进去拧。打不开？换一把。

后来徒弟成了名匠。他的弟子问他秘诀，他说："一个锁匠的水平，不取决于他的手有多巧，而取决于他的钥匙串上挂了多少把钥匙，以及他是否知道每把钥匙能开什么锁。"

🔮 揭示：LangGraph 中的 Tool — LLM 不是靠背诵解决问题，而是选择合适的工具去执行
锁→ 用户需要解决的任务
钥匙串→ tools 列表
每把钥匙→ 一个 @tool 函数
钥匙上的标签→ tool 的 name + docstring（LLM 看到的说明书）
徒弟选钥匙→ LLM 在 tool_calls 中决定调用哪个工具
钥匙插进去拧→ ToolNode 执行函数
锁开了→ ToolMessage 返回结果给 LLM

🔍 侦探的方法

L04 · ReAct 循环

福尔摩斯不是神。他有一个方法。

每接到一个案子，他不会直接给出答案。他会先推理："这个脚印说明凶手身高六尺；窗台上的烟灰是印度雪茄，说明凶手有海外关系；壁炉里的未烧完的纸片是一张船票——我需要确认是哪条航线。"

然后他行动：派人去码头查乘客名单。

然后他观察：拿到了名单，发现一个名字和烟灰的来源对上了。

然后他继续推理："这个人的住所应该在码头附近——我再去查附近的租房记录。"

就这样，推理 → 行动 → 观察 → 推理 → 行动 → 观察，一直循环，直到他站起来说："华生，我知道凶手是谁了。"

华生问他为什么不像苏格兰场那样凭直觉办案。福尔摩斯说："直觉只能解决你已经见过一百遍的案子。对于真正的谜题，你需要一套方法。"

🔮 揭示：ReAct 循环 — Reasoning + Acting，Agent 的核心执行模式
推理→ LLM 节点：分析 State，决定是否调用工具
行动→ 工具执行节点：调用外部 API / 函数
观察→ ToolMessage 追加到消息列表
循环→ 工具结果 → 再次进入 LLM 节点
"我知道凶手是谁了"→ LLM 不再返回 tool_calls，返回最终文本
苏格兰场凭直觉→ 没有工具的纯 LLM（只会生成文本）

📜 书记官的账本

L05 · Checkpoint 持久化

古罗马的元老院有一个规矩：每一场辩论，不管持续多久，书记官必须在每个人发言结束后——不是辩论结束后——立即把刚才的发言一字不差地记在羊皮纸上。如果辩论被暴风雨或敌人的号角打断，下次开会时，书记官翻开羊皮纸，从上次中断的地方继续记录。

有一年，罗马遭受瘟疫。元老院讨论抗疫措施，辩论持续了整整三天。第三天傍晚，一场地震摧毁了元老院的穹顶。辩论中断。

一个月后，元老们在临时帐篷里重开辩论。书记官打开羊皮纸——最后一条记录是："保民官提议封城三十天，执政官尚未回应。"于是辩论从这个断点继续，仿佛地震从未发生。

千年之后，一个叫 LangGraph 的程序继承了书记官的智慧：每一个 super-step 都是一次"发言结束"，都是一条新记录。

🔮 揭示：LangGraph Checkpoint 持久化 — 每个 super-step 自动存档，崩溃后从断点恢复
羊皮纸→ Checkpointer 的存储层
每次发言后记录→ 每个 super-step 自动产生 checkpoint
书记官→ Checkpointer（InMemorySaver / SqliteSaver）
羊皮纸编号→ thread_id（区分不同的辩论/会话）
翻开旧羊皮纸继续→ graph.invoke(..., config) 用相同 thread_id 恢复
翻阅历史记录→ get_state_history()

🌉 桥上的收费站

L06 · Human-in-the-Loop

有一座桥，连接两个王国。桥中间设了一个收费站。不是每辆车都要停——空车直接过，普通货车自动扣费，但运输火药、运送囚犯、装载金银的车，必须在收费站停下，由守桥官亲自检查货物清单，签字盖章后才能放行。

有人问守桥官："为什么不让所有车都自动通过？"

守桥官说："三个月前，一辆声称'运输粮食'的货车过桥时突然爆炸。如果那辆车通过的不是一座有收费站的桥，而是无人看守的通道——对面就是王宫。"

"那为什么不让所有车都停下来检查？"

"那样桥就堵死了。每天只有三辆车装的可能是炸药。为了三辆车让三千辆车排队，不值得。"

🔮 揭示：LangGraph interrupt() — 不是所有操作都需要人工审批，只对高风险操作暂停
桥→ 图的执行流程
收费站→ interrupt() 调用点
守桥官→ 人类审批者
危险货物清单→ interrupt() 的 payload
签字盖章→ Command(resume=True)
空车/普通货车直接过→ 不需要 interrupt 的安全操作

📨 信使与碎片

L07 · Streaming 流式输出

两个国王通信。起初，国王 A 让书记官写完一整封信，封好，交给信使，骑马三天送到国王 B 手中。

国王 B 经常急得跳脚——第一句话可能就回答了核心问题，但他要等三天才能看到。

后来，一个聪明的书记官发明了一种新的通信方式：他写好一句话，就把这张纸片交给等在一旁的快马信使；写第二句，第二个信使出发；写第三句，第三个信使出发……

此后，国王 B 在第一天就能收到开头几句，先知道大意；第二天收到中间几段；第三天收到结尾和签名。虽然总时间没有变短，但国王 B 的感受完全不同——"我不再是在黑夜里等待一封不知何时到达的信了。"

🔮 揭示：LangGraph Streaming — 边生成边输出，用户无需等待整个响应完成
整封信封好再送→ graph.invoke() 阻塞等待全部完成
写一句送一句→ stream_events() token-by-token
快马信使→ stream.messages 迭代器

🏰 城墙上的四道门

L08 · 容错与错误处理

一座要塞修了四道城墙。外敌侵犯时：

第一道门设在最外层——弓箭手朝敌人射三箭，如果敌人仍然冲过来，门自动关闭。这是给"小股骚扰"准备的，大部分敌人在第一道门就被挡住了。

第二道门是陷阱——敌人冲进来后，地面塌陷。但守军会喊话："投降，给你吊上来。"敌人如果愿意投降，还能活。

第三道门是一间密室——守军把信鸽放出去，向友军求救："我们遇到麻烦了，你们有什么建议？"信鸽带回来的指令可能是"再坚持一天"或"撤退"。这道门不是自动的，需要人的决定。

第四道门是最后一道。守军只在墙上刻一句话："下次修墙的时候，记住今天为什么被攻破。"然后将此事记录在案，撤退。

要塞从未被攻陷。不是因为城墙坚不可摧，而是因为这四道门的设计——大部分问题在前两道门就解决了，能冲到第四道门的敌人，一定是有值得学习的新攻击方式。

🔮 揭示：LangGraph 四层错误处理策略
第一道门→ RetryPolicy（自动重试瞬时故障如网络超时）
第二道门→ LLM 感知错误（工具返回错误信息，LLM 自行调整）
第三道门→ interrupt() 暂停，等待人工决策
第四道门→ 降级回复 + 日志记录，防止静默失败

🏙 城中之城

L09 · Subgraph 子图

长安城太大，一个京兆尹管不过来。于是天子设"里坊制"：全城分为一百零八坊，每坊有自己的坊正。坊内的事情——治安、市场、清洁——坊正自己处理。坊和坊之间，通过城门和街道连接。京兆尹只管理"跨坊"的事务：主干道的秩序、城门开关、税收统算。

最巧妙的是：每个坊的内部管理方式可以完全不同。东市的管理规则是"日出开市、日落闭市"，西市是"按货物种类分区经营"，胡人聚居的坊有自己的头人和规矩。京兆尹不需要知道东市几点开门——他只需要知道"给东市的税收指标是多少，实际交了多少"。

🔮 揭示：LangGraph Subgraph — 父图只管理子图的输入输出接口，子图内部独立自治
长安城→ 父图（Parent StateGraph）
每个坊→ 子图（Subgraph）
坊正→ 子图的 compiled graph
城门和街道→ 父子图 State 映射（不同 schema 时）
京兆尹只看跨坊事务→ 父图只关心子图的输入输出
每坊管理方式不同→ 子图可以有不同的 State 和内部结构

🐝 蜂巢的议会

L10 · Multi-Agent 协作

蜂巢里住着三种蜜蜂。采集蜂负责找花、采蜜；建筑蜂负责修蜂巢、调节温度；守卫蜂负责警戒外敌。起初，蜂后给每只蜂直接下达指令——但蜂巢太大了，蜂后的指令传到最远的蜂巢角落要半小时。

于是蜂巢进化出一个新制度：蜂后只做两件事——听取报告、分配大方向。她说："最近蜜源少了，采集蜂增加一倍，建筑蜂减半。"然后就不再说话了。采集蜂的队长自己去分配今天去哪片花田，建筑蜂的队长决定今天修哪面墙。

遇到特殊情况——比如一只采集蜂发现了一片从未见过的巨大花田——她会直接飞到建筑蜂的队长那里说："给我派一百只蜂，那片花田需要人手。"不需要经过蜂后。

🔮 揭示：LangGraph Supervisor + Swarm 多 Agent 模式
蜂后分配大方向→ Supervisor 模式（中央调度）
采集蜂直接找建筑蜂→ Swarm / Handoff 模式（Agent 间直接交接）
每种蜂→ 一个独立的 Agent（编译后的子图）
队长→ 每个 Agent 的 LLM 决策节点

🦅 将军的侦察兵

L11 · Send API + 并行执行

将军要在茫茫群山中找到敌军主力。他派出五名侦察兵，每人负责一个方向——东、南、西、北、中——同时出发。他自己坐在中军帐里，桌上铺着一张空白地图。

侦察兵们每隔一炷香放一只信鸽回来。将军把信鸽带来的情报标注在地图上：东面是悬崖，无人；北面有马蹄印，约三百骑兵；西面有炊烟，可能是营地……

两名侦察兵一去不回。但将军没有等他们——他根据已收到的三份情报做出了判断：敌军在北面。他率军向北，打了胜仗。战后第三天，失踪的侦察兵才回来——原来遇到了山洪，绕了一大圈。

一位文官问将军："为什么不派一个侦察兵把所有方向都看完再回来？"将军答："一个人看完五个方向要五天。五个人同时看各自的方向只要一天。而且就算有两人生病了，剩下三个人的情报也够我用。"

🔮 揭示：LangGraph Send API — 动态并行分发，Reducer 自动合并结果
五名侦察兵同时出发→ Send API 创建 N 个并行 worker
信鸽带回情报→ 每个 worker 返回结果
将军在地图上标注→ Reducer（如 operator.add）自动合并结果
两名侦察兵不回来也不影响→ 并行 worker 失败不阻塞其他 worker
一个人看完要五天→ 顺序执行 vs 并行执行的时间差

⚒ 从打铁铺到兵工厂

L12 · 生产部署

从前有一个铁匠，手艺精湛。他在村里开了间打铁铺，自己拉风箱、烧炉子、打铁、淬火、交货。全村的人都来找他打农具。

王国要发动战争，要求铁匠一个月内打造一万把剑。铁匠算了算——自己一天最多打五把，一万把要两千天。他需要一百个铁匠。

于是他做了几件事：

第一，他把自己打剑的过程写成一本手册。不是"怎么打一把好剑"的笼统描述，而是每一步——炉温多少度、铁烧多久、锤几下、淬火用什么油、冷却多久——精确到新来的学徒也能看懂。

第二，他建了一条流水线——一个铁匠只负责烧炉，一个只负责锤打，一个只负责淬火……

第三，他在每个工位后面放了一个本子——每做完一把剑就记一笔。如果某个工位出了问题，不用从头再来，翻开本子就知道问题出在哪个环节。

第四，他在城里设了一个展示间，官员们可以随时来看：现在打造了多少把剑、每把剑的质量如何、哪个工位最快。

一个月后，一万把剑如期交付。铁匠被封为兵部尚书。

🔮 揭示：LangGraph 从开发到生产部署的完整路径
打铁铺→ 本地开发（hello_graph.py）
