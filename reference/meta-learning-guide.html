🗿 一、六个核心思维模型
🗿 1. Six Core Mental Models

LangGraph 专家的思考方式与初学者不同——不是因为知道更多 API，而是因为他们用这六个模型理解所有问题。前五个来自学习实践，第六个来自面试反馈。面试官正是用这些模型来设计考题，区分"会调 API"和"真懂 Agent 架构"。
LangGraph experts think differently from beginners — not because they know more APIs, but because they use these six models to understand every problem. The first five come from learning practice; the sixth comes from interview feedback. Interviewers use these very models to design questions that distinguish "API callers" from those who "truly understand Agent architecture."

模型 1：图即程序
Model 1: Graph as Program
Graph as Program

初学者想：
Beginner thinks:
LangGraph 是一个库，有一些类和方法。
LangGraph is a library with some classes and methods.

专家想：
Expert thinks:
我的程序不是顺序脚本，是一个有向图。控制流从"按行执行"变成"按边流动"。
My program is not a sequential script — it's a directed graph. Control flow shifts from "execute line by line" to "flow along edges."

自检：
Self-Check:
你能在 30 秒内把一段业务流程画成节点+边的图吗？
Can you sketch a business process as a nodes + edges diagram in 30 seconds?

模型 2：State 是唯一真相来源
Model 2: State is the Single Source of Truth
Single Source of Truth

初学者想：
Beginner thinks:
State 就是一个 dict，存点变量。
State is just a dict — store some variables in it.

专家想：
Expert thinks:
节点之间不直接调用、不传参、不共享全局变量——只通过读写 State 交互。这像一个团队共用一块白板。
Nodes don't call each other directly, don't pass parameters, don't share globals — they interact solely by reading and writing State. It's like a team sharing a single whiteboard.

自检：
Self-Check:
你给一个节点新增字段时，是否先想好了"谁写？谁读？用什么 reducer？"
When adding a new field to a node, do you first decide "Who writes? Who reads? What reducer?"

模型 3：确定性框架包裹非确定性推理
Model 3: Deterministic Framework Wrapping Non-Deterministic Reasoning
Duality

初学者想：
Beginner thinks:
AI Agent 很随机，不可靠。
AI Agents are random and unreliable.

专家想：
Expert thinks:
图的路径（走向哪个节点、何时结束）是确定的；LLM 在节点内的输出是不确定的。我用图结构"驯服" LLM 的随机性。
The graph's path (which node to go to, when to end) is deterministic; the LLM's output within a node is non-deterministic. I use the graph structure to "tame" the LLM's randomness.

自检：
Self-Check:
当 Agent 行为异常时，你先怀疑图结构还是先怀疑 prompt？
When the Agent behaves unexpectedly, do you suspect the graph structure first or the prompt first?

模型 4：控制力在连接处，不在执行处
Model 4: Power Lies at the Boundaries, Not in Execution
Power at Boundary

初学者想：
Beginner thinks:
LangGraph 的核心价值在节点里。
LangGraph's core value is inside the nodes.

专家想：
Expert thinks:
节点里只是普通 Python。真正的控制力在节点之间——条件边、interrupt、retry、checkpoint、stream。20% 时间写节点，80% 时间设计边。
Inside a node is just plain Python. The real control lies between nodes — conditional edges, interrupt, retry, checkpoint, stream. 20% of time writing nodes, 80% of time designing edges.

自检：
Self-Check:
你最近写的 Agent，节点间逻辑比节点内逻辑更复杂吗？
In your most recent Agent, is the inter-node logic more complex than the intra-node logic?

模型 5：持久化是默认，不是附加
Model 5: Persistence is the Default, Not an Add-On
Persistence by Default

初学者想：
Beginner thinks:
先写好 Agent，需要时再加"保存"功能。
Build the Agent first, add "save" functionality later if needed.

专家想：
Expert thinks:
每个 super-step 都自动存档。程序崩溃不丢状态——这是框架默认行为。用"写数据库应用"而非"写脚本"的思维来设计。
Every super-step is auto-saved. Crash without losing state — this is framework-default behavior. Design with the mindset of "writing a database application" not "writing a script."

自检：
Self-Check:
你写 node 时，是否默认假设"它可能在返回前被中断、再从断点恢复"？
When writing a node, do you assume it "might be interrupted before returning, then resumed from the breakpoint"?

🔗 面试官的评估框架（第一性原理）— 五个模型如何映射到面试

面试官只需要验证三件事，恰好对应你学的五个模型：
The interviewer only needs to verify three things, which precisely map to the five models you're learning:

    Can you BUILD?
     —
    代码实现、API 设计 → 模型 1(图即程序) + 模型 2(State 设计)

  2. 你能 REASON 吗？

    Can you SHIP?

🔥 专家在吵什么
🔥 What the Experts Are Debating

Graph API vs Functional API

声明式便于可视化、团队协作 vs 命令式更自然、代码更少。官方说"混着用"，但实际团队往往选一边。
Declarative is better for visualization and team collaboration vs imperative is more natural with less code. Official stance says "mix them," but teams in practice tend to pick a side.

多 Agent 是否必要
Are Multi-Agent Systems Necessary?

分工协作各展所长 vs 一个大模型就够了，多 Agent 增加延迟和复杂度。
Division of labor, each playing to its strengths vs one large model is enough; multi-Agent adds latency and complexity.

LangGraph 是否过度抽象
Is LangGraph Over-Abstracted?

提供了恰好需要的原语 vs FastAPI + 消息队列也能做到，还更灵活。
Provides exactly the right primitives vs FastAPI + message queues can do it too, and with more flexibility.

MCP vs A2A 边界在哪 🆕
Where is the MCP vs A2A Boundary? 🆕

MCP 管 Agent↔Tool，A2A 管 Agent↔Agent。但实践中两个 Agent 通过 MCP 互调能替代 A2A 吗？社区争议：MCP 是为被动工具设计的，不适合有自主推理的 Agent 间协作。
MCP handles Agent↔Tool, A2A handles Agent↔Agent. But in practice, can two Agents calling each other through MCP replace A2A? Community debate: MCP was designed for passive tools, not suited for collaboration between Agents with autonomous reasoning.

长上下文是否杀死 RAG 🆕
Will Long Context Kill RAG? 🆕

1M token 上下文窗口 + Prompt Caching，是否让 RAG 变得多余？反方：成本（O(n²) Attention）、Lost in the Middle 效应、缓存失效意味着 RAG 仍有不可替代的价值。
1M token context windows + Prompt Caching — does this make RAG redundant? Counterargument: cost (O(n²) Attention), Lost in the Middle effect, cache invalidation — meaning RAG still has irreplaceable value.

Deep Agents SDK vs 裸 LangGraph 🆕
Deep Agents SDK vs Bare LangGraph 🆕

Deep Agents 提供了 planning、subagents、filesystem 开箱即用，但增加了一层抽象。直接写 LangGraph StateGraph 更灵活但更繁琐。选哪个取决于任务复杂度。
Deep Agents provides planning, subagents, and filesystem out of the box, but adds a layer of abstraction. Writing raw LangGraph StateGraph is more flexible but more tedious. The choice depends on task complexity.

🟢 基础篇测试（L01-L04 结束后）
🟢 Foundation Test (After L01-L04)

Q1：节点函数没有 return 语句时会怎样？
Q1: What happens when a node function has no return statement?

点击查看答案
Click to View Answer

不会报错。隐式返回 None，LangGraph 将其解释为"无状态更新"，State 保持不变。但如果你依赖这个节点的输出来更新 State，后面的节点可能拿到意料之外的值。
No error occurs. It implicitly returns None, which LangGraph interprets as "no state update" — State remains unchanged. But if you depend on this node's output to update State, downstream nodes may receive unexpected values.

Q2：Annotated[list, operator.add] 和普通 list 在并行节点场景下有什么区别？
Q2: What is the difference between Annotated[list, operator.add] and a plain list in a parallel node scenario?

点击查看答案
Click to View Answer

普通 list：后完成的节点覆盖先完成的节点的值（替换）。
Annotated + operator.add：两个节点的输出自动合并（追加）。并行时多个节点同时返回列表，reducer 决定如何合并而非覆盖。
Plain list: the later-completing node overwrites the earlier node's value (replacement).
Annotated + operator.add: both node outputs auto-merge (append). When parallel nodes return lists simultaneously, the reducer decides how to merge rather than overwrite.

Q3：条件边路由函数返回了一个未在映射字典中列出的值，图会怎么处理？
Q3: What happens when a conditional edge routing function returns a value not listed in the mapping dictionary?

点击查看答案
Click to View Answer

抛出 ValueError。映射字典必须是穷尽的——每个可能的返回值都要有对应目标节点。这也是为什么建议在条件边函数中设兜底分支（else → 默认节点）。
Throws a ValueError. The mapping dictionary must be exhaustive — every possible return value must have a corresponding target node. This is why it's recommended to include a fallback branch (else → default node) in conditional edge functions.

Q4：bind_tools 之后，LLM 收到的 API 请求里多了什么？
Q4: After bind_tools, what extra content appears in the LLM's API request?

点击查看答案
Click to View Answer

多了 tools 字段，包含每个工具的 JSON Schema：name、description、parameters（类型+描述）。LLM 据此决定是否调用工具——它不是"看到" Python 函数，而是看到一份"工具说明书"。
An additional tools field appears, containing the JSON Schema for each tool: name, description, parameters (types + descriptions). The LLM uses this to decide whether to call a tool — it doesn't "see" the Python function, it sees a "tool manual."

Q5：为什么 state["messages"][-1] 就能判断是否需要调用工具？
Q5: Why can state["messages"][-1] determine whether a tool call is needed?

点击查看答案
Click to View Answer

因为 LangGraph 的消息流是线性的——最后一条消息一定是上一步产生的。如果上一步是 LLM，最后一条就是 AIMessage，它的 tool_calls 属性说明 LLM 决定调工具。如果是 ToolMessage（工具返回结果），则 tool_calls 为 None，应该再给 LLM 处理。
Because LangGraph's message flow is linear — the last message is always the one produced by the previous step. If the previous step was the LLM, the last message is an AIMessage, and its tool_calls attribute indicates the LLM decided to call a tool. If it's a ToolMessage (tool result), then tool_calls is None, and it should be fed back to the LLM for processing.

🟡 专业篇测试（L05-L08 结束后）
🟡 Professional Test (After L05-L08)

Q6：同一个 thread_id 调两次 invoke()，第二次的消息是追加还是替换？
Q6: If you call invoke() twice with the same thread_id, are the second call's messages appended or do they replace?

点击查看答案
Click to View Answer

追加。因为 State 中的 messages 使用了 operator.add reducer。每次 invoke 的消息会追加到同一个 thread 的消息历史后面。如果换一个新的 thread_id，则从空列表开始。
Appended. Because the messages field in State uses the operator.add reducer. Each invoke's messages are appended to the same thread's message history. If you use a new thread_id, it starts from an empty list.

Q7：interrupt() 和 retry_policy 的区别？什么场景需要两个一起用？
Q7: What's the difference between interrupt() and retry_policy? When should you use both together?

点击查看答案
Click to View Answer

retry_policy 是给机器用的（网络超时自动重试），interrupt() 是给人用的（等待人工决策）。两者可以配合：先 retry 3 次，如果全失败，进 interrupt() 让人决定是继续重试还是放弃。
retry_policy is for machines (auto-retry on network timeouts); interrupt() is for humans (await human decision). They can work together: retry 3 times first; if all fail, enter interrupt() to let a human decide whether to keep retrying or give up.

Q8：子图的 token 流在哪里？
Q8: Where is the subgraph's token stream?

点击查看答案
Click to View Answer

在 stream.subgraphs[*].messages 中。v3 的 typed projections 把子图的消息流隔离在独立的 subgraph 命名空间下，不会和父图的消息混在一起。
In stream.subgraphs[*].messages. v3's typed projections isolate the subgraph's message stream under an independent subgraph namespace, preventing it from mixing with the parent graph's messages.

Q9：interrupt() 暂停后重启进程，能用原 thread_id 恢复吗？
Q9: After interrupt() pauses and the process restarts, can you resume with the original thread_id?

点击查看答案
Click to View Answer

取决于用的是什么 checkpointer。InMemorySaver → 不行（内存丢失）。SqliteSaver / PostgresSaver → 可以（持久化到磁盘/数据库，进程重启后从 checkpoint 恢复）。这就是为什么生产环境必须用持久化 checkpointer。
Depends on which checkpointer is used. InMemorySaver → no (memory is lost). SqliteSaver / PostgresSaver → yes (persisted to disk/database, restored from checkpoint after process restart). This is why production environments must use persistent checkpointers.

🔴 专家篇测试（L09-L12 结束后）
🔴 Expert Test (After L09-L12)

Q10：父子 State 有同名字段但子图直接 add_node 挂载，消息怎么合并？
Q10: Parent and child State share a same-named field, but the subgraph is mounted via add_node — how do messages merge?

点击查看答案
Click to View Answer

子图共享父图的 State 通道。如果子图的 messages 也用了 operator.add reducer，子图中产生的消息会自动追加到父图的 messages 列表中。这是因为底层 Pregel 运行时共享同一个 State 实例。
The subgraph shares the parent graph's State channel. If the subgraph's messages also use the operator.add reducer, messages produced in the subgraph are automatically appended to the parent graph's messages list. This is because the underlying Pregel runtime shares the same State instance.

MCP
MCP
（Model Context Protocol）：Agent 需要调用工具/数据源时。它是一个 Agent 和它的工具之间的"USB-C 接口"——标准化工具接入。
A2A（Agent-to-Agent）：多个 Agent 需要互相委派任务时。它是 Agent 之间的"通讯协议"——标准化 Agent 协作。

MCP (Model Context Protocol): When an Agent needs to call tools/data sources. It's the "USB-C interface" between an Agent and its tools — standardized tool integration.
A2A (Agent-to-Agent): When multiple Agents need to delegate tasks to each other. It's the "communication protocol" between Agents — standardized Agent collaboration.
Key distinction: Does the other party have autonomous reasoning capability? A tool doesn't (MCP is sufficient); another Agent does (use A2A). The interviewer wants to hear this distinction logic, not recited definitions.

Q17：为什么说"长上下文不能替代 RAG"？给出 3 个面试官无法反驳的理由。
Q17: Why do we say "long context cannot replace RAG"? Give 3 reasons that an interviewer cannot refute.

点击查看答案
Click to View Answer

① 成本：1M token 的 Attention 是 O(n²)，成本是 4K 的数百倍。Prompt Caching 只能缓解，不能消除。
② Lost in the Middle：实验证明 LLM 对长文本中间部分的 recall 率显著低于开头和结尾。塞得越多，检索越差。
③ 更新频率：知识库内容频繁更新时，每次都重新打包进 context 无法利用缓存，而 RAG 只更新变化的文档即可。
① Cost: Attention for 1M tokens is O(n²), costing hundreds of times more than 4K. Prompt Caching only mitigates, it doesn't eliminate.
② Lost in the Middle: Experiments show LLM recall of middle sections in long texts is significantly lower than for beginnings and endings. The more you cram in, the worse the retrieval.
③ Update frequency: When knowledge base content changes frequently, repacking everything into context each time can't leverage caching, whereas RAG only needs to update the changed documents.

为什么寓言比直接解释更有效？
Why Are Fables More Effective Than Direct Explanation?

Anthropic 专家 Amanda Askell 在 2025 年 4 月访谈中指出：人类用故事传递知识已经有一万年了。当你在听一个故事时，大脑中多个区域同时激活——画面、情绪、因果关系——形成了一个比"背诵定义"丰富得多的记忆网络。过一周，你忘了技术定义，但忘不了故事。故事中隐含的逻辑会自动触发概念回忆。
Anthropic expert Amanda Askell pointed out in her April 2025 interview: Humans have been passing down knowledge through stories for ten thousand years. When you listen to a story, multiple brain regions activate simultaneously — imagery, emotion, causality — forming a memory network far richer than "reciting a definition." A week later, you'll forget the technical definition, but you won't forget the story. The logic embedded in the story automatically triggers concept recall.
